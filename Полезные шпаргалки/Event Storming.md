**Event Storming** – это нетехнический воркшоп, в результате которого получается схематическое описание системы. ES позволяет моделировать систему с точки зрения происходящих в ней процессов.

На ES отвечают на 3 вопроса:
1. как ведёт себя система для разных пользователей?
2. из каких процессов состоит система?
3. как процессы связаны между собой?

В ES используются стикеры 6 разных цветов, у каждого из которых своя роль.

| Цвет стикера | Описание                                                                                                                                                                                                |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Желтый       | Актор, запускающий команду. Может быть живым (администратор, клиент) или неживым (приложение).                                                                                                          |
| Голубой      | Действие **в настоящем времени**, которое совершает система. Оно должно отвечать на вопрос "Что произойдет?", а не на вопрос "Как именно это произойдет?". Пример: отметить заказ собранным.            |
| Оранжевый    | Событие **в прошедшем времени** (его нельзя отменить), которое является результатом выполнения команды.<br>Пример: account registered.                                                                  |
| Фиолетовый   | Policy или **бизнес-условие**, которое должно быть соблюдено для выполнения команды. Пример: для того, чтобы отправить заказы в доставку, их должно накопиться не менее 3 штук.                         |
| Розовый      | Внешние системы (черные ящики), с которыми взаимодействует наша.<br>Пример: система оплаты, система отправки SMS-сообщений.                                                                             |
| Зеленый      | Модель для чтения (read model) с данными о части системы, без которой актор **не может принять решение о выполнении команды**.<br>Пример: чтобы сделать checkout необходимо знать, что лежит в корзине. |
| +Белый       | Для комментариев.                                                                                                                                                                                       |

Пример Event Storming для системы чекаута состоящей из 3 процессов: 
1. оформление заказа в магазине, 
2. сборка заказа,
3. пополнение склада от поставщика, если товар начинает заканчиваться.
