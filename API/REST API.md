**REST** – это стиль проектирования [[API (Application Programming Interface)|API]] (набор архитектурных принципов) для сложных распределенных интернет-сервисов.

Стиль проектирования REST – это гибрид, который создан на основе трех архитектурных парадигм:
1. **Client-Server**. Приложение разбивается на две части: клиент с UI и сервер с бизнес-логикой и БД.
2. [[Stateless и Stateful сервисы|Stateless]]. Сервер не должен делать записи о состоянии клиента. Каждый новый запрос должен быть сформирован так, будто раньше клиент с сервером не общались.
   Если клиент запросил страницу 1, то в новом запросе он не может запросить следующую страницу, подразумевая страницу 2, так как сервер не запоминает, какая страница была предыдущей (не хранит состояние клиента).
3. **Cache**. Сервис помечает свои ответы, как кэшируемые, а клиент в свою очередь сохраняет их у себя в памяти. При повторном запросе одних и тех же данных клиент заберет их из кэша. Кэширование снижает нагрузку на сервер и увеличивает отзывчивость.

**Унификация интерфейса** – это еще один ключевой принцип REST, который вводит ряд архитектурных ограничений, который делает интерфейс общения между клиентом и сервером единообразным (это важно для поддержки и расширения системы):
1. **Идентификация ресурса** (текстовый документ, JSON, картинка итд). Каждый ресурс должен иметь стабильный уникальный идентификатор URI (Unified Resource Identifier).
2. **Манипуляции ресурсом через представление**. Клиент не может напрямую изменять данные на сервере. Для изменения данных клиент должен отправить серверу свое полное видение данных, то есть предоставить измененный ресурс целиком. Сервер на своей стороне принимает решение о том принимать это изменение или нет.
3. **Самоописываемые сообщения.** Сообщения, которые клиент получает от сервера, должны содержать в себе всю необходимую информацию для того, чтобы клиент понял информацию и смог ее интерпретировать (нужен конекст).
4. **Гипермедиа (href), как двигатель состояния приложения**. Сервер должен подсказывать клиенту, какие следующие шаги он может предпринять дальше.

**Принцип многоуровневости**. Важно понимать, что в архитектуре REST могут существовать не только уровни клиента и сервера. Так, например, между клиентом и сервером может быть установлен прокси-слой (или балансировщик нагрузки), отвечающий за авторизацию. Таким образом для клиента прокси будет выполнять роль сервера, а для сервера роль клиента. Слоев может быть сколько угодно, но каждый из компонентов знает лишь о существовании ближайшего компонента к нему.
